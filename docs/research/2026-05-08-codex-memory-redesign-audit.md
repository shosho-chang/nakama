## Section 4 — ASSUMPTION PUSH-BACK

Evidence read before writing this section: I read the proposal’s Decision A, Decision B, Open Decision G, and “Specific items I want the panel to challenge”; I compared them to `CLAUDE.md`, current workflows, ADR-002, ADR-016, and ADR-021’s warning about rubber-stamped HITL.

Decision A, `memory-trunk`: reject. It is overengineered for a solo developer. The proposal’s chosen flow says memory commits go to `memory-trunk`, push directly, and fast-forward into `main` weekly at `E:\nakama-design\docs\research\2026-05-08-memory-system-redesign-v1.md:98`. That creates a permanent second trunk, delayed visibility, branch discipline, fast-forward failure modes, and extra hooks. A cron job or direct-to-main auto-commit with `paths-ignore: memory/**` achieves the same CI isolation without inventing a long-lived branch. If branch protection blocks direct pushes, solve that rule directly; do not add a branch model before proving exemption is impossible.

Decision B, L3 confirm-mode: reject as underspecified. The proposal says “Do NOT auto-write. Instead: list candidate memories ... ask user which to save” at `E:\nakama-design\docs\research\2026-05-08-memory-system-redesign-v1.md:171`. Functionally, that is just “do not auto-write” unless a concrete UI or CLI prompt is implemented. The proposal does not specify what process catches “清對話,” who surfaces the confirm prompt, how multiple Claude windows avoid duplicate prompts, or how a confirmed choice becomes a bounded file change. ADR-021 already warns that HITL can become rubber-stamping at `E:\nakama\docs\decisions\ADR-021-annotation-substance-store-and-brook-synthesize.md:38`. Hard-disable session-end writes until there is an implementation spec.

Open challenge 4, Codex integration: the Codex framing is real but not load-bearing for the full redesign. The current `memory/claude/` name is a problem if Codex is a peer agent, and `CLAUDE.md` currently points only to that directory at `E:\nakama\CLAUDE.md:165`. But the natural fix is a namespace prefix such as `memory/codex/` plus either `memory/shared/` or an `agent:` field in frontmatter. ADR-002 already anticipated an `agent` field at `E:\nakama\docs\decisions\ADR-002-memory-system-redesign.md:103`. Codex does not require `memory-trunk`, `_archive/`, a new index generator, and L3 confirm-mode all at once.

Specific challenge verdicts: 1. `memory-trunk` is overengineered; use direct-to-main plus workflow filtering first. 2. L3 confirm-mode is not meaningfully different from “don’t auto-write” without a concrete prompt implementation. 3. The raw 297-file count is not the problem; stale session/project files and noisy feedback are. 4. ## Section 5 — ALTERNATIVES NOT CONSIDERED

Evidence read before writing this section: I compared the proposal’s constraints C1-C6 against the current repo-backed memory model, the SQLite maintenance code, workflows, ADR-002, and the Sandcastle/share-nothing guidance.

Alternative 1, memory as a git submodule: This works for C1 cross-platform sync because memory remains git-backed and can be cloned on Windows and Mac. It also isolates memory commit history from application history. But it makes C3 Sandcastle harder: sub-agents need submodule checkout, write access, branch state, and push credentials separately. It also weakens C6 backward compatibility unless `memory/claude/` stays in place or is replaced by a submodule at exactly that path. Not recommended for this repo right now.

Alternative 2, memory as SQLite synced via git: This fits the existing infrastructure better than the proposal admits. `shared/memory_maintenance.py` already operates on SQLite rows, and `shared.state` already creates the `memories` table. A single binary blob commit per session would directly reduce the “264 commits touching `memory/`” noise by consolidating writes. The downside is severe: binary DB merge conflicts are unresolvable without a custom merge driver or append-only export/import protocol. Viable only if a custom merge driver or deterministic JSONL export is added; otherwise risky under C3 parallel agents.

Alternative 3, Codex integration via MCP memory protocol: This is more future-proof than file conventions if Codex can read memory through the same protocol as Claude or other agents. ADR-002 already listed “MCP Memory Server” as a future Phase 4 option at `E:\nakama\docs\decisions\ADR-002-memory-system-redesign.md:275`. But it is premature for Phase 0 because it assumes Codex supports MCP reads at session start and that all relevant execution surfaces can be configured consistently. The proposal’s file namespace is less elegant but more immediately controllable.

Alternative 4, minimum viable cleanup: This is the strongest short-term alternative. Rewrite `feedback_conversation_end.md` to stop auto-write/commit/push, add `paths-ignore: memory/**`, and periodically archive or delete `project_session_*` files. It directly addresses C5 signal-to-noise and C4 no-review/no-CI overhead with zero architectural change. It does not solve Codex namespace cleanliness, but it does not need to. Namespacing new Codex writes can be added separately without migrating 289 tracked files.

## Section 6 — FINAL VERDICT

Evidence read before writing this section: I read the full proposal, verified the required file counts and schema claims, inspected current workflows and `CLAUDE.md`, checked ADR-002/ADR-016/ADR-021, and reran the 30-day memory commit count.

Verdict: approve with modifications. The proposal correctly identifies the core disease: the current trigger turns “清對話” into automatic memory write, index edit, commit, and push. It is also right to distinguish durable memory from session logs. I agree with the proposal’s phrase “Memory ≠ session log” at `E:\nakama-design\docs\research\2026-05-08-memory-system-redesign-v1.md:82`. But I reject the unnecessary machinery around that insight.

Phase 0 should not land as drafted. Split it into two PRs. PR A should be file hygiene only: archive or delete stale `project_session_*` and noisy feedback files according to an explicit retention rule, rewrite `feedback_conversation_end.md` so session end does not auto-write, and add `paths-ignore: memory/**` to workflows. PR B should be schema and namespace only: add an `agent:` field to frontmatter for new writes, create `memory/codex/` as a Codex namespace, and update the index format. Do not combine these with a new branch model.

Drop `memory-trunk` entirely. Use direct-to-main with a `paths-ignore: memory/**` workflow filter instead. If branch protection blocks direct memory commits, document that with GitHub settings evidence and solve the exemption directly. Do not create a permanent second trunk as a workaround for an unverified rule.

Drop L3 confirm-mode until it has a concrete implementation spec. “Ask user which to save” is not enough. Specify the command surface, prompt owner, concurrency behavior, file-write path, and commit behavior before treating it as a design decision.

Do not require migration of the 289 existing tracked files. The current schema is mixed, and at least one sampled feedback file lacks `type`. New schema should apply only to new writes. Migration can be optional and batch-confirmed later after validation tooling exists.
