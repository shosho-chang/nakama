### Section 3 — NUMERICAL / FACTUAL CLAIMS

FrankyEvolveV1 §7's weekly synthesis context estimate of 50k tokens is not realistic for the current source list if the implementation feeds only picked items plus compact ADR and issue snapshots. `E:/nakama/config/ai_news_sources.yaml` lists 12 RSS or Atom feeds, and `news_digest.py` truncates each candidate summary to 600 characters for curation and each scored summary to 1500 characters for scoring. The daily digest output contains selected scored items, not full source articles. With the current 5-8 picks, seven days of selected item summaries, scores, URLs, and reasons should be far below 50k tokens. After §2 Fork 4 changes the pick range to 8-12, the weekly selected item context still lands in a manageable band if it uses digest artifacts rather than raw feed content. The 50k number only becomes plausible if the weekly call injects a 30-day ADR/MEMORY/plans corpus directly. That makes the estimate a retrieval design choice, not a consequence of `ai_news_sources.yaml`. Fix §7 by splitting the estimate into "weekly picked news tokens" and "retrieved internal-context tokens."

The marginal cost claim of "~$3/month" is therefore under-specified, not proven. The spec prices "one LLM call/week" with "~50k tokens" at "~$0.40/week" without naming the model or input/output token split. The existing `weekly_digest.py` cost code proves Nakama can calculate costs from `api_calls` using `shared.pricing.calc_cost`, but it does not validate the proposed new call shape. Require an implementation note that logs the weekly synthesis call into `api_calls` with `model`, `input_tokens`, `output_tokens`, cache read/write tokens, and operation id. Do not freeze the budget until one dry run records actual token usage.

FrankyEvolveV1 §6 underestimates S2. "RAG corpus + Relevance dimension" is not one slice; split it into two slices. First build the retrieval substrate: corpus selection, chunking, embedding, reranking, refresh schedule, cache invalidation, bilingual evaluation fixtures, and failure behavior when retrieval returns nothing. Then build the scoring change: `news_score.md` adds Relevance, `news_digest.py` injects retrieved context, the pick gate changes to require `relevance >= 3`, rendering includes relevance, and tests recalibrate old examples. Combining these creates an oversized slice with two different failure modes. Do not mix retrieval correctness with scoring-policy calibration.

### Section 4 — ASSUMPTION PUSH-BACK

Q1 BGE-M3 cross-domain retrieval is not established as robust from the permitted evidence. The ADR-020 path is missing, so its status and evaluation cannot be quoted. The framing is wrong: the decision is not "BGE-M3 versus LLM-as-retriever"; the decision is "prove bilingual cross-domain retrieval on Nakama artifacts before using it in a gate." Do a small labeled eval with English news titles and summaries mapped to Traditional Chinese ADR/MEMORY snippets. Ship BGE-M3 only if recall at the chosen `k` is high enough to support a hard `relevance >= 3` gate.

Q2 the "zero proposals is legal" prompt mitigation is insufficient. LLMs satisfy shape pressure, and §4 gives the weekly synthesis a hard output range of 1-3 proposals in the pipeline diagram. Change that to 0-3 everywhere. Add a hard quality gate before issue creation: only create an issue when at least two independent news items support the same trend, or one item directly maps to a named open issue or ADR assumption. Send weak singletons to the weekly vault page only.

Q3 LLM-judged "architectural lock-in" is a high confirmation-bias zone. Use a deterministic positive list plus LLM explanation. Automatically mark `panel_recommended: yes` when the proposal references an ADR, changes an agent contract, introduces a new persistent dependency, changes a storage schema, changes HITL boundaries, or changes Slack/GitHub automation privileges. The LLM can add reasons; it must not be the only classifier.

Q4 GitHub trending Python with `agent|llm|mcp|claude` is a vibecoded-demo and star-farming magnet. Do not feed raw trending into the same trust tier as official blogs and known release feeds. Add repo-age, release, README, license, commit-history, issue-activity, and maintainer checks before curation. Better: start with the two Anthropic repo feeds and curated awesome-list diffs, then add GitHub trending only as an experimental source with a lower trust label.

Q5 success metrics are overclaimed. Cost reduction metrics can be verified through `api_calls`; qualitative metrics cannot be verified by the tables read in `weekly_digest.py`. Require every proposal to declare `metric_type: quantitative | checklist | human_judged`, `baseline_source`, and `verification_owner`. For human-judged metrics, the retrospective reports decision quality and owner verdict, not fake numeric precision.

Q6 Franky is overloaded under ADR-001. Split the evolution loop into a provisional `Sengoku` agent or formally amend ADR-001 to make Franky both maintenance and evolution. The cleaner architecture is a new agent that consumes Franky's digests and internal context, then writes proposal candidates. Franky should keep source fetching, health, CVE, and operational reporting.

Q7 Relevance weight 1.3 is a guess. Run a one-week dry run before freezing weights. Score the same candidate set with old four-dimensional scoring and new five-dimensional scoring, then compare picks with owner judgments. Do not ship a hard `relevance >= 3` gate until false negatives are reviewed, because the gate can suppress genuinely important model/tool releases that lack an obvious current Nakama mapping.

### Section 5 — ALTERNATIVES NOT CONSIDERED

Alternative (a): treat Franky's evolution loop as a passive newsletter that Brook or Robin pulls from, not an issue creator. Verdict: use this as the first production mode. It preserves ADR-001 better than auto-created GitHub issues because Franky remains a digest producer rather than a planning actor. It also preserves HITL discipline because the owner or a downstream agent decides what becomes work. The downside is weaker closure: newsletters get ignored. Fix that by adding explicit "promote to issue" commands or owner checkboxes in the weekly digest. Start passive, measure promotion rate, then allow issue creation only after quality is proven.

Alternative (b): dispatch synthesis to a Codex or Sandcastle subagent instead of another Franky LLM call. Verdict: use this for architectural proposals, not for ordinary weekly summaries. FrankyEvolveV1 §3 already says Franky must not run the panel. A subagent is a better boundary for higher-risk synthesis because it can inspect a bounded context package, produce an audit-style recommendation, and leave Franky's operational code simpler. The cost is orchestration complexity and latency. That cost is acceptable for proposals that touch ADRs, schemas, dependencies, or agent contracts. Keep low-risk weekly clustering as a cheap LLM call; dispatch high-risk candidates to a subagent before issue creation or mark them as panel-required without creating execution work.

Alternative (c): skip RAG entirely and inject a fixed `MEMORY.md` snapshot into the score prompt. Verdict: use this for S1/S2 dry runs, not as the final design. It beats premature RAG because the current spec has not proven ADR-020 infrastructure availability, bilingual retrieval quality, or chunk policy. A fixed snapshot is cheap, deterministic, debuggable, and enough to test whether Relevance improves curation. It fails once the snapshot goes stale and it misses ADR-level nuance. Use a generated weekly `franky_context_snapshot.md` containing active priorities, current ADR assumptions, and top open issues. Replace it with RAG only after retrieval evals beat the snapshot baseline.

The spec also skips a fourth practical alternative: a two-stage proposal inbox. Stage one writes all synthesis candidates to `KB/Wiki/Digests/AI/Weekly-YYYY-WW.md` with machine-readable frontmatter. Stage two creates GitHub issues only for candidates explicitly promoted by the owner or by a deterministic policy after repeated signals. Verdict: this is the best modification to the proposed issue flow. It keeps the closed loop, avoids issue queue pollution, and gives retrospectives a clean denominator: candidates, promoted proposals, shipped proposals, and successful proposals.

### Section 6 — FINAL VERDICT

Approve with modifications.

1. Change FrankyEvolveV1 §2 Fork 2 and §4 from automatic weekly issue creation to a two-stage proposal inbox: weekly digest first, GitHub issue only after owner promotion or a strict repeated-signal rule. This keeps `needs-triage` meaningful and prevents Franky from becoming an unapproved planning actor.

2. Split FrankyEvolveV1 §6 S2 into two slices: retrieval substrate and Relevance scoring calibration. The current S2 mixes infrastructure, prompt changes, ranking math, pick gates, and tests. That is too broad for the 2-4 hour estimate and hides retrieval failure behind scoring changes.

3. Amend FrankyEvolveV1 §5 Q6 and ADR planning language to address ADR-001 directly. Either create provisional `Sengoku` for evolution synthesis or update ADR-001's Franky row. Do not implement a role expansion while the accepted ADR says Franky is System Maintenance.

4. Replace FrankyEvolveV1 §7 cost and token estimates with measured dry-run numbers. Log weekly synthesis calls through `api_calls`, record model and token split, and separate news-context tokens from internal-context tokens. The current "~$3/month" number is not grounded enough for approval.

5. Add a metrics store before implementing FrankyEvolveV1 §2 Fork 5 monthly retrospectives. The existing operational tables support cron, alert, backup, incident, and cost summaries; they do not store proposal lifecycle or success verification data. Without `proposal_id`, baseline, post-ship value, status, and verification method, the retrospective becomes narrative theater instead of feedback.
