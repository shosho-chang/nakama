### 4.2 🟡 Non-blockers (4)

#### NB1 — Error handling coverage incomplete

**Location:** Plan §7 R2; Plan §4.3 ebook step 1 / inbox step 5
**Issue:** Plan §7 R2 only specifies behavior for `extract_metadata` `MalformedEPUBError` ("return None from `_resolve_book`; log warning"). Two adjacent failure paths are not covered:
- `book_storage.read_book_blob(book_id, lang=...)` may raise `BookStorageError` / `FileNotFoundError` (DB row exists but blob is missing on disk)
- `extract_frontmatter(read_text(path))` may raise on malformed inbox markdown frontmatter

**Why it matters:** Without explicit guidance, the implementing agent will pick a behavior (raise vs silently return None vs partial ReadingSource). Inconsistency with §7 R2's "return None + warn" spirit is likely.

**Concrete fix:**
- Extend §7 R2 to cover all blob-read / fm-parse failures uniformly.
- Add explicit catch-and-return-None instructions in §4.3 ebook step 1 (after `read_book_blob`) and inbox step 5 (after `extract_frontmatter`).

**Severity rationale:** Not a blocker — agent can implement reasonably without this. But shipping without it produces undocumented edge-case behavior that will need a follow-up PR.

---

#### NB2 — bilingual-only inbox primary_lang ambiguity

**Location:** Plan §4.3 inbox step 7 case (b)
**Issue:** For inbox case (b) bilingual-only (no original sibling on disk), Plan §4.3 step 7 says `primary_lang` is from the bilingual sibling's frontmatter `lang:` field.

In practice, `shared/translator.py` writes the bilingual sibling. The `lang:` field there may be:
- The **source** language (what the bilingual was translated FROM)
- The **target** language (what 修修 reads in zh)
- Empty / "bilingual"

Convention is not pinned. If translator writes `lang: zh-Hant` (target), `primary_lang` reflects the target, which contradicts the schema docstring ("BCP-47 short lang of the **evidence/original-language** content").

**Why it matters:** Downstream `#511` reads `primary_lang` and may make wrong evidence-language assumptions in case (b). The `evidence_reason="bilingual_only_inbox"` flag does signal "primary_lang is best-effort here," but only if `#511` knows to interpret the flag that way.

**Concrete fix:**
- Add a §4.3 step 7 (b) note: "`primary_lang` for bilingual-only is best-effort — translator's `lang:` convention is not guaranteed to match evidence-track semantics. Downstream slices reading `primary_lang` should consult `evidence_reason` and treat case (b) lang as low-confidence."
- Or add a §7 R7 explicit risk entry.

**Severity rationale:** Doesn't break any v2 contract on its own. Risks downstream misinterpretation.

---

#### NB3 — `metadata.lang_pair` legacy passthrough may seduce downstream re-derivation

**Location:** Plan §4.3 ebook step 7 (metadata pass-through)
**Issue:** ebook resolution emits `metadata = {isbn, published_year, "lang_pair": book.lang_pair, "original_metadata_lang": ...}`.

Q1 says #509 must not derive `primary_lang` from `lang_pair`. v2 doesn't — it correctly uses `BookMetadata.lang`. But by exposing `lang_pair` in `metadata`, the registry hands a deprecating field to downstream consumers (`#511`, `#513`, `#514`), who may re-derive language from it instead of trusting `ReadingSource.primary_lang`.

**Why it matters:** Q1 spirit is "lang_pair is not the contract for language." Pass-through arguably violates the spirit even if not the letter.

**Concrete fix path A:** Drop `lang_pair` from metadata entirely — downstream can read it from `Book` directly if absolutely needed.
**Concrete fix path B:** Keep but add a docstring tag: "metadata.lang_pair: legacy reference only. Do NOT consume to derive language semantics; use ReadingSource.primary_lang."

**Severity rationale:** Q1 is followed in #509's own logic. Risk is purely about influencing downstream behavior. Either fix path is cheap.

---

#### NB4 — Test 13 leaves implementation choice unresolved

**Location:** Plan §4.4 test 13; Plan §7 R6
**Issue:** Test 13 (`test_resolve_inbox_empty_annotation_slug`) reads:
> "Either raises a controlled exception OR returns ReadingSource with `annotation_key == ""` and a TODO/log warning — pick one explicitly in implementation, do not silently produce malformed key."

§7 R6 lean: "raise `ValueError`."

But the lean is not promoted to §5 self-imposed gates, so the implementing agent retains freedom to choose either path. Both paths can pass test 13 (which only requires "controlled" behavior, not a specific shape).

**Why it matters:** `#510` Reading Overlay will need to handle whatever #509 chose. If #509 raises and #510 expects None-or-empty, #510 will need a try/except wrapper. Locking the behavior in #509 saves this.

**Concrete fix:** Add §5 self-imposed gate: "Empty annotation_slug → raise `ValueError` (per §7 R6 lean)." Tighten test 13 description to assert the raise path specifically.

**Severity rationale:** Not a contract violation in v2. Cross-slice contract clarity issue.
