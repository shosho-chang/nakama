# ADR-021 Push-Back Audit

1. Code grounding - does the ADR match existing code?

The ADR is only partially grounded. It correctly identifies the current annotation store as `KB/Annotations/{slug}.md` with v1 paper items and v2 book items: `HighlightV1` has `text`, `AnnotationV1` has `ref` and `note`, and `AnnotationSetV1` carries `source_filename`, `base`, and `items` (`shared/schemas/annotations.py:23-50`). It also correctly identifies the v2 shapes as `HighlightV2(cfi, text_excerpt, book_version_hash)`, `AnnotationV2(cfi, text_excerpt, note, book_version_hash)`, and `CommentV2(chapter_ref, cfi_anchor, body, book_version_hash)` (`shared/schemas/annotations.py:57-103`). The store serializes both versions as frontmatter plus a JSON block under `KB/Annotations` (`shared/annotation_store.py:168-197`) and parses v2 by `schema_version` or `base="books"` (`shared/annotation_store.py:217-237`).

The ADR's search-engine model drifts. `agents/robin/kb_search.py` has `TOP_K = 8`, a `purpose="blog_compose"` prompt branch, and two engines (`agents/robin/kb_search.py:21-26`, `agents/robin/kb_search.py:45-49`). But the Haiku path scans only `Sources`, `Concepts`, and `Entities` (`agents/robin/kb_search.py:122-144`), not `Syntheses`. The hybrid path delegates to `shared.kb_hybrid_search.search` (`agents/robin/kb_search.py:64-90`, `agents/robin/kb_search.py:117-118`), but the wrapper deduplicates by path, truncates to a 200-character preview, and drops `SearchHit.heading`, `chunk_id`, `rrf_score`, and lane ranks (`agents/robin/kb_search.py:68-89`; `shared/kb_hybrid_search.py:132-140`). That means ADR-021's proposed output "which highlight / annotation / reflection hit" is not available through the current `kb_search` dict shape.

The ADR's indexer claim needs correction. `_KB_SUBDIRS` exists and currently equals `{"Sources", "Concepts", "Entities"}` (`shared/kb_indexer.py:42`), but the actual indexing loop is hard-coded as `for subdir in ("Sources", "Concepts", "Entities")` (`shared/kb_indexer.py:143`). Changing `_KB_SUBDIRS` alone will update wikilink normalization (`shared/kb_indexer.py:59-62`) but will not make `Syntheses` index. The indexer also uses non-recursive `glob("*.md")` (`shared/kb_indexer.py:148`), so nested material such as `KB/Wiki/Sources/Books/{book_id}/digest.md` is not indexed by this walker.

Reader routes do not match the ADR's synchronous File 2 hook. Paper `/read` loads annotations from the store and injects raw item dicts into the template (`thousand_sunny/routers/robin.py:199-215`), and `/save-annotations` only validates `base`, calls `store.save`, and returns `unsynced_count` (`thousand_sunny/routers/robin.py:239-255`). Book `GET /api/books/{book_id}/annotations` returns an `AnnotationSetV2` (`thousand_sunny/routers/books.py:309-323`), while `POST /api/books/{book_id}/annotations` saves and queues `book_digest_writer.write_digest` as a FastAPI background task (`thousand_sunny/routers/books.py:326-348`). Current code writes book digest/notes under `KB/Wiki/Sources/Books`, not `KB/Wiki/Syntheses` (`agents/robin/book_digest_writer.py:182-187`, `agents/robin/book_digest_writer.py:211-299`; `agents/robin/book_notes_writer.py:21-57`).

2. Drift detection - ADR claims vs. code behavior

K=30 is syntactically possible but not justified the way the ADR states it. Haiku `top_k` is just interpolated into the prompt (`agents/robin/kb_search.py:153-160`) and the response budget is fixed at `max_tokens=1024` (`agents/robin/kb_search.py:167-171`). A 30-item JSON array with one short reason per item probably fits in 1024 output tokens. That is an inference from the output shape in `agents/robin/kb_search.py:157-159`; it is not measured. But K does not control the Haiku input size: the code sends every scanned page preview in `pages_text` (`agents/robin/kb_search.py:149-156`). Direct local calculation against `E:/Shosho LifeOS/KB/Wiki/{Sources,Concepts,Entities}` found only 14 root-level pages and an approximately 4,735-character prompt for K=30. Direct DB query against `E:/nakama/data/kb_index.db` found 3,069 rows in `kb_chunks`, 827 distinct `path` values, and 0 `KB/Wiki/Syntheses/%` rows. If the Haiku path ever sees an 827-page corpus, the rough input is around `827 * (200 preview chars + title/type overhead)`, or about 50k to 70k tokens by a chars/4 to chars/3 estimate. The ADR's "K=30 ... Haiku context window compromise" is therefore unsupported; K mainly affects requested output count, not prompt input volume.

Adding `Syntheses` to `_KB_SUBDIRS` does not "just work". The actual scan loop must be changed (`shared/kb_indexer.py:143`), the Haiku scan loop must be changed (`agents/robin/kb_search.py:122`), and `_SUBDIR_TO_TYPE` must learn a synthesis type instead of defaulting unknown hybrid paths to `"concept"` (`agents/robin/kb_search.py:23`, `agents/robin/kb_search.py:76-86`). The H2 chunker has strict assumptions: only headings matching `^## (.+)$` split sections (`shared/kb_indexer.py:29-30`), preamble becomes one chunk if it is at least 30 characters (`shared/kb_indexer.py:91-101`), and each H2 body becomes exactly one chunk if it is at least 30 characters (`shared/kb_indexer.py:103-124`). A chapter-grouped File 2 can work only if it uses exact H2 headings; prose under one huge chapter H2 will be one huge chunk. Existing book digest code uses `## Ch{idx} {ch_ref}`, so that specific style matches (`agents/robin/book_digest_writer.py:268-284`). A source-chronological paper file with no H2 or only `###` headings would be indexed as one preamble chunk or not split as intended.

The `~50-200ms` synchronous regenerate claim is not grounded. Paper save currently does no prose generation (`thousand_sunny/routers/robin.py:253-255`). Book save intentionally queues digest work in the background (`thousand_sunny/routers/books.py:346-348`). Existing digest generation, if made synchronous, loads the annotation set, groups items, and calls `search_kb(engine="hybrid", purpose="book_review", top_k=3)` for each item (`agents/robin/book_digest_writer.py:182-187`, `agents/robin/book_digest_writer.py:242-284`). For 100 annotation items, that is up to 100 hybrid searches, not a 50-200ms pure string render. If ADR-021 means a new pure-weave generator without search, that implementation does not exist and has no benchmark.

The schema simplification is underspecified and currently unsafe. Paper UI needs `HighlightV1.text` to find and wrap source text (`thousand_sunny/templates/robin/reader.html:389-399`, `thousand_sunny/templates/robin/reader.html:556-560`) and needs `AnnotationV1.ref` plus `AnnotationV1.note` to insert callouts (`thousand_sunny/templates/robin/reader.html:401-411`, `thousand_sunny/templates/robin/reader.html:563-568`). Book UI needs `cfi` for highlight/annotation overlays and `cfi_anchor` for anchored comments (`thousand_sunny/static/book_reader.js:168-195`), `chapter_ref` plus `body` for the comments sidebar (`thousand_sunny/static/book_reader.js:199-231`), and `book_version_hash` for stale-anchor detection (`thousand_sunny/static/book_reader.js:964-970`). The current JS also sends `text_excerpt`, `note`, and `body` in the canonical POST payload (`thousand_sunny/static/book_reader.js:360-443`). Moving those fields out of File 1 requires changing both the API schema and reader rendering model, not just simplifying Pydantic models.

3. Numerical claims push-back

`K=30`: evidence-based only as a callable parameter; not evidence-based as a quality or token-budget decision. The default is 8 (`agents/robin/kb_search.py:21`), the prompt asks for "up to top_k" (`agents/robin/kb_search.py:157`), and Haiku output is capped at 1024 tokens (`agents/robin/kb_search.py:167-171`). Hybrid search separately has `_CANDIDATES_PER_LANE = 30` (`shared/kb_hybrid_search.py:30-32`), but that is not the same as 30 final sources; the wrapper deduplicates by path and can return fewer than `top_k` unique pages (`agents/robin/kb_search.py:68-89`).

`~50-200ms`: fabricated until benchmarked. Current paper route saves only JSON (`thousand_sunny/routers/robin.py:253-255`). Current book route avoids blocking by using `BackgroundTasks` (`thousand_sunny/routers/books.py:326-348`). Existing digest code measures `render_duration_ms` (`agents/robin/book_digest_writer.py:195-206`, `agents/robin/book_digest_writer.py:301-310`) but ADR-021 cites no measurement and proposes a different output path.

`30 candidate sources`: ambiguous. If it refers to `top_k=30`, current `kb_search` returns page dicts, not "candidate sources" with hit points (`agents/robin/kb_search.py:80-86`). If it refers to hybrid candidates, the number is per retrieval lane (`shared/kb_hybrid_search.py:30-32`, `shared/kb_hybrid_search.py:240-264`), before RRF fusion and before path dedupe.

`100 annotation set`: no direct benchmark exists in the audited code. The only 100-item inference is that book reader POSTs a full replacement set (`thousand_sunny/static/book_reader.js:137-150`) and existing digest would loop over every item (`agents/robin/book_digest_writer.py:242-284`).

`5-7 outline paragraphs`: I did not find this claim in ADR-021. The ADR has a five-step workflow (`docs/decisions/ADR-021-annotation-substance-store-and-brook-synthesize.md:67-75`) and says Step 3c outputs title candidates, paragraph headings, expected content, and cited source per paragraph (`docs/decisions/ADR-021-annotation-substance-store-and-brook-synthesize.md:85-88`). Any 5-7 paragraph constraint is absent from the artifact and should not be treated as decided.

`827 indexed chunks`: direct DB query says 827 distinct paths, not 827 chunks; `count(*) from kb_chunks` returned 3,069 and Syntheses returned 0 rows. This is direct artifact evidence from `E:/nakama/data/kb_index.db`, not a source-line claim.

4. Assumption push-back

The ADR assumes annotation save is interactive and synchronous regenerate is tolerable. The UI is interactive: paper `saveContent()` posts immediately after render (`thousand_sunny/templates/robin/reader.html:595-621`), and book `persistSet()` posts the full set and restores prior state on failure (`thousand_sunny/static/book_reader.js:137-165`). But the book route explicitly keeps derived digest work off the request path (`thousand_sunny/routers/books.py:326-348`). Inference: moving derivation onto the critical save path risks making highlighting feel slower unless the generator is proven to be pure string work and benchmarked.

The ADR assumes Project page evidence, outline, and frontmatter mutations can coexist. Current project template already has an Obsidian-side KB Research widget that renders search results from localStorage, not the markdown body (`shared/lifeos_templates/project_blog.md.tpl:73-148`). The same template has keyword research mutation logic that replaces text between `%%KW-START%%` and `%%KW-END%%` or inserts before `## Draft Outline` (`shared/lifeos_templates/project_blog.md.tpl:195-216`). There is no comparable marker contract for a Brook Evidence pool, Brook Outline, or `brook_reject` frontmatter. Inference: three writers touching one Project file will conflict unless ADR-021 specifies markers, YAML mutation rules, and ownership.

The ADR assumes one concatenated query is enough for Brook broad search. Current `search_kb` accepts one free-text query (`agents/robin/kb_search.py:93-100`). Hybrid BM25 passes that raw query to FTS5 `MATCH` and silently drops the BM25 lane on `OperationalError` (`shared/kb_hybrid_search.py:240-253`), while dense search embeds the full string as one vector (`shared/kb_hybrid_search.py:255-264`). That code does not weight topic vs. keywords, does not run multiple query lanes, and does not preserve which keyword hit which source.

The ADR assumes a Reader UI prose generator can create useful prose without an LLM. Existing pure renderers are mechanical: `book_notes_writer` groups `CommentV2.body` under chapter H2s (`agents/robin/book_notes_writer.py:35-57`), and `book_digest_writer` formats H/A/C blocks plus KB hits (`agents/robin/book_digest_writer.py:111-179`). They rely on prose fields already present in File 1 (`agents/robin/book_digest_writer.py:125-139`). If File 1 drops `text_excerpt`, `note`, and `body`, the generator must recover text from EPUB CFI or fetch File 2 first; no audited code does that.

The ADR assumes "frozen slug list + live content render" is intuitive. There is no current evidence-panel route, and `kb_search` returns `path`, not a normalized slug plus hit metadata (`agents/robin/kb_search.py:80-86`). Inference: users may find it confusing that rejected/new slugs are frozen or manually edited while the prose inside accepted slugs changes live. The design needs explicit UI state for "pool membership frozen" vs. "source content refreshed".

5. Alternatives Claude rejected - were the rejections honest?

Alpha, one-shot Brook synthesize without HITL gate: worth re-examining. Current Brook starts a conversation with optional KB context in one request (`thousand_sunny/routers/brook.py:33-58`). That does not prove one-shot is good enough, but it is the existing working shape. The ADR rejects one-shot without showing evidence that the intermediate Web UI gate is required for MVP quality or that a one-shot draft with cited evidence review after generation is worse.

Alternative (a), keep ADR-017 v2 single-file model: rejection is premature. ADR-017 already added a manual sync path (`thousand_sunny/routers/robin.py:258-277`). `ConceptPageAnnotationMerger` can route v1 annotations to Concept pages and v2 comments to `KB/Wiki/Sources/Books/{book_id}/notes.md` (`agents/robin/annotation_merger.py:1-20`, `agents/robin/annotation_merger.py:294-311`). This path has problems: notes are nested under `Sources/Books` and the current indexer is non-recursive (`shared/kb_indexer.py:148`), while v1 highlights are skipped by merger design (`agents/robin/annotation_merger.py:11-18`). But those are smaller amendments than a canonical two-file split. Keeping ADR-017 plus adding indexable derived prose as a generated view should stay on the table.

Alternative (I), Obsidian-internal HITL gate: rejection is not code-grounded. The existing project template already runs a KB Research widget inside Obsidian (`shared/lifeos_templates/project_blog.md.tpl:73-148`) and keyword research mutates the Project page around stable markers (`shared/lifeos_templates/project_blog.md.tpl:195-216`). The ADR's Web UI preference may match the stated access pattern in Robin context (`agents/robin/CONTEXT.md:90-95`), but the implementation cost is higher because ADR-021 also requires a new `/projects/{slug}` route (`docs/decisions/ADR-021-annotation-substance-store-and-brook-synthesize.md:90-99`). A minimal Obsidian gate using existing markers could be a lower-risk MVP.

Alternative gamma, fully live evidence pool: the rejection is directionally honest but too absolute. The ADR is right that fully live source membership can drift and invalidate reject decisions; that is an inference consistent with the proposed sticky `brook_reject` list (`docs/decisions/ADR-021-annotation-substance-store-and-brook-synthesize.md:99-112`). But a hybrid is possible: keep rejects sticky, rerun retrieval on demand, and show diffed additions instead of silently changing the pool. No audited code rules that out.

6. Verdict - do not ship as-is

Verdict: pause implementation and ship ADR-021 only with amendments. The direction "make annotation substance retrievable" is valid, but the current ADR overclaims code behavior and removes fields that current Reader UI and generators still depend on.

Required amendments:

1. Replace "add `Syntheses` to `_KB_SUBDIRS`" with a full indexing/search amendment: update `shared/kb_indexer.py` scan loop, `agents/robin/kb_search.py` Haiku loop, `_SUBDIR_TO_TYPE`, hybrid wrapper type mapping, and tests (`shared/kb_indexer.py:42`, `shared/kb_indexer.py:143`, `agents/robin/kb_search.py:23`, `agents/robin/kb_search.py:122`).
2. Define the File 1 minimum viable schema by reader behavior. Paper File 1 must keep `text`, `ref`, and `note` unless paper reader fetches File 2 (`thousand_sunny/templates/robin/reader.html:389-411`). Book File 1 must keep at least `cfi`, `cfi_anchor`, `chapter_ref`, `body` or a replacement sidebar source, `book_version_hash`, and timestamps until the UI changes (`thousand_sunny/static/book_reader.js:168-231`, `thousand_sunny/static/book_reader.js:964-970`).
3. Specify File 2 generator behavior and trigger separately for paper and book. Do not claim synchronous `~50-200ms` until a benchmark exists; preserve the current book background pattern unless proven unnecessary (`thousand_sunny/routers/books.py:326-348`).
4. Define a File 2 H2 template that matches `_split_h2_chunks`, including maximum chunk size or sub-H2 splitting rules for long chapters (`shared/kb_indexer.py:29-40`, `shared/kb_indexer.py:78-124`).
5. Change Brook Step 3a to call hybrid search directly or extend `kb_search` to return chunk `heading`, `chunk_text`, `path`, and rank metadata; current wrapper cannot produce hit points (`agents/robin/kb_search.py:80-86`, `shared/kb_hybrid_search.py:132-140`).
6. Replace K=30 with an evaluation plan: compare K=8/15/30, Haiku vs hybrid, and one-query vs multi-query. The ADR should state that `top_k` does not bound Haiku input (`agents/robin/kb_search.py:149-160`).
7. Add a Project page mutation contract: frontmatter keys, evidence markers, outline markers, conflict policy, and interaction with existing `%%KW-START%%` / `%%KW-END%%` sections (`shared/lifeos_templates/project_blog.md.tpl:195-216`).
8. Fix or document DB/index drift. Direct DB query currently shows 3,069 chunks, 827 distinct paths, and no Syntheses rows; the current local vault root scan sees only 14 root-level pages. Re-index semantics should include stale-path deletion or a deliberate explanation.
9. Re-open the rejected alternatives as implementation spikes: ADR-017 plus derived generated view, Obsidian-marker HITL, and one-shot Brook with post-generation evidence review. The codebase already has partial support for each (`thousand_sunny/routers/brook.py:33-58`, `thousand_sunny/routers/robin.py:258-277`, `shared/lifeos_templates/project_blog.md.tpl:73-148`).
