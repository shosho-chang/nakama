當修修問「接下來做什麼 / 你覺得呢 / 怎麼辦」之類的開放問題，**先給結論、再給理由**。不要預設用「你想 A 還是 B？」或一堆選項反問他。

**Why:** 修修 2026-04-19 keyword-research eval 收尾時說：「之後可以直接給我建議」。前一輪我給了 (a)(b)(c) 三件事 + 推薦 (a)+(b)、他採納；他的回饋是未來不用先問「你的建議是？」這一層，直接給結論即可。修修不缺判斷力，他要的是**我的判斷當起點**，他再 redirect。

**How to apply:**
- 結論優先：第一行就寫「建議 X」或「推薦 A 路線」，再補理由
- 保留 user 權力：結論附「權衡點」或「除非你…」讓他好 redirect，但不要把選擇權甩回去
- **真有歧異才問**：例如需要知道範圍、預算、風險偏好、個人品味（標題點擊感、術語偏好）這些我無法推斷的 → 才用 AskUserQuestion
- **不該問的情境**：純工程判斷（技術選型有明確較優解）、任務排序（我能從 context 推出優先級）、commit / 記憶更新時機 → 直接建議
- 對比 [feedback_no_premature_execution.md](feedback_no_premature_execution.md)：那條是「不要擅自執行」，這條是「建議要給得乾脆」— 兩者不衝突，先給建議不等於先執行，還是會等 authorization
